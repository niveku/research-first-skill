# Writing & Content Reference

Domain-specific guidance for writing, documents, templates, strategies, and content tasks (reports, PRDs, briefs, OKRs, status reports, go-to-market plans).

## Step 1: Context detection

Check these before searching:

- Existing style guides, brand guidelines, editorial handbooks
- Previous docs in the project (a past PRD, past quarterly review, past brief)
- Preferred doc format (Markdown, Google Docs, Notion, Confluence)
- Template library the team already uses (Notion workspace, Google Drive folder, internal wiki)
- Audience (internal, external, executive, engineering, marketing)

**Domain-fit guidance** - work within what exists; propose revisions only if there's a real gap:

- Team has a brand style guide -> follow it. Propose a revision only if it has a real gap for this task.
- Project has an existing PRD / brief / report template -> build on it. Suggest a new format only if the existing one structurally doesn't fit.
- Company uses Google Workspace -> propose a Docs/Sheets solution first. Suggest a new tool only if it's a clear step-change in value.
- Team uses Notion -> use Notion's template gallery; don't recommend an external tool unless Notion genuinely can't do it.

## Step 2b: Web search patterns

- `"free {report/brief/strategy} template {industry}"`
- `"{document type} framework best practices 2025"` (e.g., "OKR template", "PRD template", "GTM strategy template")
- `"site:notion.so/templates {topic}"` - Notion template gallery
- `"site:canva.com/templates {topic}"` - Canva design templates
- `"{document type} example site:github.com"` - real-world examples from open-source projects (PRDs, RFCs, SOPs)

Skip the MCP layer (Step 2a) entirely for pure writing tasks - it's not relevant unless the workflow involves a data pipeline or automation.

## Step 2c: Reusable patterns and templates

- **Document frameworks**: STAR method (situation/task/action/result), MECE (mutually exclusive, collectively exhaustive), Pyramid Principle, SBAR (situation/background/assessment/recommendation).
- **Established templates**: Amazon 6-pager, Shape Up pitch, Google design docs, RFC (IETF style), Product Requirements Doc (Lenny's Newsletter has canonical examples).
- **Brand templates**: project-specific if they exist, or industry-standard (Mailchimp, Notion, Stripe all have public style guides worth referencing).

## Step 5: Execution guidance for writing

- Use the found template as the base structure.
- Adapt tone and content to the project's style guide if one exists.
- Cite sources or frameworks used ("adapted from Amazon's 6-pager format", "based on Shape Up's pitch template").
- Don't blindly apply a template - remove sections that don't fit and add project-specific ones.

## Key resource lists (writing)

**Templates and frameworks:**
- [Notion Template Gallery](https://notion.so/templates) - docs, wikis, project trackers, strategies
- [Canva Templates](https://canva.com/templates) - design, presentations, reports
- Google Workspace templates - Docs, Sheets, Slides starter templates
- [Miro](https://miro.com/templates/) / [FigJam](https://www.figma.com/community/figjam) - process maps, retrospectives, brainstorming frameworks
- [Process.st](https://www.process.st) - SOP and workflow templates for business processes

**Canonical document references:**
- Amazon's 6-pager format (widely documented in blogs and Jeff Bezos's shareholder letters)
- Basecamp's Shape Up book (free online) - pitches, betting tables
- Lenny's Newsletter - PRD / roadmap / strategy canonical examples
- Google's design doc template (public write-ups from ex-Googlers)

**Forums for content/writing opinions:**
- Reddit: r/writing, r/marketing, r/content_marketing, r/ProductManagement
- Substack / Medium - use bias filter, prefer authors with verifiable track records
- Twitter/X - good for finding templates shared by practitioners (search "PRD template", "OKR template")
